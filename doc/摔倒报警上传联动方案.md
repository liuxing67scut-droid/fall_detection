# 摔倒报警上传联动方案

## 1. 目标

在现有摔倒检测主流程保持不变的前提下，为系统增加“检测到摔倒后，将报警事件与当前截图上传到网站平台”的能力。

本方案只关注：

- Raspberry Pi 侧如何触发上传
- 上传节流与去重策略
- 与平台服务端的接口约定
- 联调与验收步骤

本方案 **不负责** 最终网站页面样式、数据库细节与运营文案；这些交由平台/网站实现同学继续完成。

## 2. 当前工程现状

当前项目已经具备：

- `HOG + Box`
- `YOLO11 + Box`
- `MediaPipe + Pose`
- `MediaPipe (PC) + Pose (Pi)`

并且：

- `FallDetector` 可在 `bbox` 路线下输出 `normal / suspected_fall / fall_detected`
- `PoseFallDetector` 可在 `pose` 路线下输出 `normal / suspected_fall / fall_detected`
- `PoseRemoteClient` 已可在模式 4 下通过 `HTTP + Tailscale` 与电脑端服务通信

因此，最佳插入点是：

- 在 `App::previewLoop()` 中，检测 `FallState` 是否从非 `fall_detected` 切换为 `fall_detected`
- 若发生切换，则抓取当前原始图像并提交给报警上传客户端

## 3. 推荐总体方案

推荐链路：

```text
FallDetector / PoseFallDetector
    -> state becomes fall_detected
    -> App captures current frame
    -> FallAlertClient uploads one image + one alert message
    -> Platform Server stores event and image
    -> Website displays alert record
```

建议新增独立客户端：

- `FallAlertClient`

而不是直接把报警上传混进现有 `PlatformClient`，原因：

- `PlatformClient` 当前职责更偏“定时状态上报 + 指令轮询”
- 摔倒告警是低频、事件驱动、带图片上传的行为
- 单独拆分更便于维护、联调和限流

## 4. 防止“上传爆多”的规则

### 4.1 只在首次进入 `fall_detected` 时上传

触发条件：

- 前一状态不是 `fall_detected`
- 当前状态是 `fall_detected`

不在以下情况上传：

- `suspected_fall`
- `fall_detected` 持续状态的每一帧
- `normal`

### 4.2 冷却时间

新增配置：

```yaml
fall_alert_cooldown_ms: 30000
```

含义：

- 两次告警上传之间至少间隔 `30` 秒
- 即使状态短时间抖动，也不会连续刷屏

建议范围：

- `20000 ~ 30000 ms`

### 4.3 只上传一张截图

每次有效告警只上传：

- 一张当前画面截图
- 一条文本提醒
- 一组基础元数据

不做视频上传，不做连续多图上传。

## 5. 推荐接口设计

建议平台端新增专用接口：

```http
POST /api/events/fall
Content-Type: multipart/form-data
```

不建议复用现有 `/api/events/recognition`，原因：

- 语义不清晰
- 后续前端页面展示会混淆“识别事件”和“摔倒告警事件”
- 数据结构差异较大

### 5.1 请求字段

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| `source_device` | string | 设备编号，如 `pi-01` |
| `frame_id` | int | 当前帧编号 |
| `ts_ms` | int | 毫秒级时间戳 |
| `mode` | string | 当前检测模式，如 `pose:pc_mediapipe` |
| `fall_state` | string | 固定为 `fall_detected` |
| `message` | string | 报警提醒文本 |
| `fps` | float | 当前近似实时帧率 |
| `image` | file/jpeg | 当前截图 |

### 5.2 推荐提醒文本

推荐文案：

```text
检测到画面中可能有人摔倒，请及时查看。
```

更正式版本：

```text
系统检测到画面中可能存在摔倒情况，请及时确认。
```

建议前端最终显示：

- 标题：`摔倒告警`
- 正文：`检测到画面中可能有人摔倒，请及时查看。`

### 5.3 响应格式

建议平台端返回：

```json
{
  "ok": true,
  "event_id": 123,
  "image_url": "/uploads/fall/2026/05/06/fall_123.jpg",
  "message": "stored"
}
```

若失败：

```json
{
  "ok": false,
  "error": "reason"
}
```

## 6. 平台服务端建议实现

平台服务端建议新增：

- `POST /api/events/fall`
- `GET /api/events/fall?limit=100`

### 6.1 图片存储目录

建议：

```text
platform_server/data/uploads/fall/
```

可进一步按日期分层：

```text
platform_server/data/uploads/fall/2026/05/06/
```

### 6.2 数据库存储建议

建议新增 `fall_events` 表，字段至少包括：

- `id`
- `source_device`
- `frame_id`
- `ts_ms`
- `mode`
- `fall_state`
- `message`
- `fps`
- `image_path`
- `created_at`

### 6.3 WebSocket 推送

建议沿用现有 `/ws` 机制，新增广播消息类型：

```json
{
  "type": "fall_event",
  "event": {
    "...": "..."
  }
}
```

这样网站前端可以实时弹出告警。

## 7. Raspberry Pi 侧代码框架

当前项目建议新增：

- `include/alert/FallAlertTypes.hpp`
- `include/alert/FallAlertClient.hpp`
- `src/alert/FallAlertClient.cpp`

### 7.1 当前骨架职责

`FallAlertClient` 负责：

- 接收截图与告警元数据
- 应用冷却时间
- 后台异步发送 HTTP multipart 请求
- 保存最近状态字符串供调试显示

### 7.2 当前不做的事

本骨架 **不负责**：

- 网站前端页面绘制
- 平台数据库建表
- 图片 URL 生成细节
- 平台权限与 token 策略最终定稿

## 8. 建议配置项

建议在 `config/app.yaml` 中加入：

```yaml
fall_alert_enabled: false
fall_alert_base_url: "http://100.87.247.58:9000"
fall_alert_event_path: "/api/events/fall"
fall_alert_device_id: "pi-01"
fall_alert_device_token: ""
fall_alert_message: "Possible fall detected in the scene. Please check immediately."
fall_alert_connect_timeout_ms: 1000
fall_alert_timeout_ms: 3000
fall_alert_cooldown_ms: 30000
fall_alert_jpeg_quality: 85
fall_alert_debug: false
```

说明：

- 默认关闭：`fall_alert_enabled: false`
- 平台端准备好后再打开
- 当前 `message` 为 ASCII 默认值，后续可改成中文

## 9. 触发时机建议

在 `App::previewLoop()` 中：

- `bbox` 路线：使用 `fall_status.state`
- `pose` 路线：使用 `pose_fall_status.state`

伪代码：

```cpp
if (previous_state != FallState::FallDetected &&
    current_state == FallState::FallDetected) {
    enqueue current frame to FallAlertClient;
}
previous_state = current_state;
```

注意：

- 上传的是原始 `frame_packet.bgr`
- 不建议上传带完整 UI overlay 的截图
- 若需要更清晰的汇报/网页效果，可后续讨论是否上传“仅带框和标签”的渲染图

## 10. 联调测试步骤

### 10.1 平台服务端

需要队友完成：

- `POST /api/events/fall`
- 能接收 multipart/form-data
- 能保存图片
- 能返回 `ok: true`

### 10.2 Pi 侧验证

验证项：

1. 未进入 `fall_detected` 时，不上传
2. 首次进入 `fall_detected` 时，上传一次
3. 持续停留在 `fall_detected` 时，不重复上传
4. 冷却时间内再次跌倒，不重复上传
5. 冷却时间后再次跌倒，可以重新上传

### 10.3 网站端验证

验证项：

1. 新事件是否能出现在列表
2. 图片是否能正常查看
3. 文本是否正确显示
4. 多次触发时是否会爆多

## 11. 当前推荐落地顺序

1. 先在 Pi 侧保留当前 `FallAlertClient` 骨架
2. 队友先实现平台端 `/api/events/fall`
3. 先用 Postman / curl 手动测平台端接口
4. 再从 Pi 端打开 `fall_alert_enabled`
5. 最后联调“真实摔倒触发 -> 平台显示图片告警”

## 12. 交付给队友时的重点说明

交付时建议明确告诉队友：

- Pi 侧已经预留好了异步上传骨架
- 需要平台端新增专用 `fall` 事件接口
- 需要支持 multipart 图片上传
- 需要服务端做落盘与事件入库
- 需要前端把 `fall_event` 作为单独告警展示，而不是混入人脸识别事件

这样分工最清晰，也最容易联调。
