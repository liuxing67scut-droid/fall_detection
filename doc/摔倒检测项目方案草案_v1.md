# 摔倒检测项目方案草案 v1

## 1. 项目目标

本项目拟使用树莓派 4B 和 USB 摄像头，完成一个面向学校室内环境的单人摔倒检测原型系统。

当前第一版目标如下：

- 场景限定在室内，如宿舍、实验室、教室
- 摄像头采用固定机位
- 只考虑单人场景
- 第一版输出以屏幕可视化为主，不先做报警联动
- 开发语言以 C++ 为主

这个版本的重点是先把完整链路跑通，而不是一开始追求复杂场景下的高精度。

## 2. 第一版边界

第一版计划做的内容：

- 采集 USB 摄像头视频流
- 在树莓派上实时显示画面
- 完成人体检测与框选
- 在条件允许时增加人体关键点或姿态可视化
- 基于规则做摔倒判断
- 在屏幕上显示当前状态，如 `normal`、`suspected fall`、`fall detected`

第一版暂不优先做的内容：

- 多人同时检测
- 复杂遮挡场景优化
- 自训练摔倒分类模型
- 语音报警、短信推送、联网通知
- 跨场景高泛化能力

## 3. 总体技术路线

当前更适合的路线不是直接训练一个完整的摔倒识别模型，而是采用“现成轻量模型 + 规则判定”的方案。

推荐拆分如下：

1. 视频采集与显示：使用 OpenCV 完成摄像头读取、画面显示和截图调试
2. 人体检测：使用轻量人体检测模型，先确认树莓派 4B 上的运行速度是否可接受
3. 姿态估计：在人体检测稳定后，再评估是否加入关键点检测
4. 摔倒判定：结合人体框比例、身体朝向、关键点高度变化、持续时间等规则进行判断

这样做的原因：

- 比自己训练模型更容易落地
- 更适合树莓派 4B 的算力条件
- 能够分阶段验证，不容易一开始就陷入模型训练和部署细节

## 4. 对齐队友工程的原则

为了便于后续和队友的人脸识别项目保持风格一致，并为未来合并预留空间，当前采用以下策略：

- 保持目录层次和模块划分风格尽量一致
- 参考对方的入口组织方式和类命名风格
- 优先对齐通用模块接口，如相机、渲染、配置读取方式
- 不直接复制与人脸识别强绑定的业务逻辑
- 在保证功能清晰的前提下，为后续合并预留兼容空间

实现策略明确为：

- 我们自己写
- 主动对齐队友工程风格和接口习惯
- 不直接整套移植参考工程

### 阶段 2：人体检测与框选

目标：

- 在视频中识别人
- 用矩形框标记人体区域
- 在画面上显示基础检测信息
- 第一版只保留 `person` 类别
- 第一版在单人模式下只保留一个主框

验收标准：

- 单人站立、走动、弯腰时能较稳定框住人体
- 树莓派端运行速度基本可接受
- 缺少模型文件时，工程仍可编译运行并给出明确状态提示

#### 阶段 2 当前选定方案

当前阶段 2 的正式路线为：

- 推理框架：`OpenCV DNN`
- 模型格式：`ONNX`
- 第一目标类别：`person`
- 运行模式：单人场景下保留一个主框

选择这条路线的原因：

- 已有工程已使用 `OpenCV + CMake + C++`
- 接入成本低于直接引入 `ncnn`
- 便于先把人体检测功能跑通
- 后续若树莓派性能不足，仍可把检测后端替换为 `ncnn`

#### 阶段 2 详细推进步骤

1. 先增加独立的 `PersonDetector` 模块，并接入工程
2. 在配置文件中加入模型路径、输入尺寸、阈值、检测间隔等配置项
3. 在主循环中接入人体检测，并在画面上绘制人体框和置信度
4. 在单人模式下，只保留一个主目标，减少后续逻辑复杂度
5. 增加基础性能控制手段，如检测间隔和结果复用
6. 在树莓派上测试检测稳定性、延迟和分辨率影响
7. 根据实际效果决定是否调低输入分辨率或降低检测频率

#### 阶段 2 当前工程策略

为了便于阶段 2 尽快落地，当前工程层面采用以下策略：

- 正式路线使用 `YOLO ONNX + OpenCV DNN`
- 在模型文件尚未准备好时，保留一个轻量 fallback，避免工程完全不可运行
- 当前阶段先做检测和画框，不做跟踪、不做关键点、不做摔倒判定

#### 阶段 2 当前性能调优方向

当前关于帧率问题的结论如下：

- `frame_fps` 表示向摄像头申请的目标帧率，不代表最终程序实际显示帧率
- 实际 FPS 会受到摄像头取流、模型推理耗时、检测间隔和分辨率等多个因素影响
- 当前优先采用小步调优，而不先引入云端协同等复杂方案

当前阶段 2 第一轮性能优化的策略：

- 在界面中显示实际 FPS 和单次检测耗时
- 优先调低检测输入尺寸，而不是立刻降低显示分辨率
- 适当增加检测间隔，降低每秒推理次数
- 在保证单人框选效果基本可用的前提下，逐步平衡准确率和流畅度

### 阶段 3：姿态或关键点可视化

目标：

- 在画面中绘制人体关键点
- 用不同颜色线条连接关节点
- 观察不同姿态下的稳定性

验收标准：

- 站立、弯腰、坐下、倒地时关键点有基本可视化效果

风险提示：

- 这一阶段比人体检测更吃性能
- 若树莓派运行过慢，可将其降级为可选模块，或延后实现

### 阶段 4：摔倒判定

目标：

- 输出正常、疑似摔倒、确认摔倒三种状态
- 不只依赖单帧结果，而是结合短时间窗口进行判断

建议规则来源：

- 人体框长宽比变化
- 身体主方向接近水平
- 头部、肩部、髋部高度变化明显
- 姿态变化在短时间内发生
- 倒地状态持续一定时间

验收标准：

- 对明显摔倒动作能给出可视化提示
- 对普通站立和缓慢行走不频繁误报

#### 摔倒检测方案对比

- 方案 A：基于框几何特征 + 时间规则
  当前主推方案。直接复用现有人体检测结果，工程量最小，最适合先做出可运行雏形。
- 方案 B：单人跟踪 + 几何规则 + 丢框保持
  当前推荐落地方案。相比方案 A，更能应对短时掉框、低置信度和姿势突变，适合作为阶段 4 的直接前置能力。
- 方案 C：姿态关键点方案
  可作为后续增强方向。理论上更适合分析摔倒姿态，但对树莓派 4B 的性能压力更大，暂不作为第一优先级。
- 方案 D：直接 fall / stand 分类模型
  暂不优先。数据集、训练与部署复杂度更高，容易把项目重心过早拉向模型调优。

#### 当前推进顺序

- 先做单人框跟踪、低置信度延续和丢框保持
- 再做基于框的摔倒规则判断
- 姿态关键点和直接分类模型作为后续增强路线，暂不先上

后续实现备注：
- 若后期需要并行保留多种检测方法，可优先采用“配置切换后端/模型路径”的轻量方式，而不是在运行主链路里加入复杂交互
- 当前阶段先保证单一路线可稳定运行，避免为了切换功能影响实时效果
- 为了减少反复手动修改 `app.yaml`，后续运行方式改为：程序启动时通过终端主菜单选择高层模式，配置文件继续保留细参数

状态机说明：
- 队友项目中的 `core/StateMachine` 更偏向应用流程状态切换，不直接复用到本项目的摔倒判定逻辑
- 当前项目在 `FallDetector` 内部维护 `normal / suspected_fall / fall_detected` 这一类领域状态，更贴合当前单人摔倒检测场景

## 6. 模型准备与下载说明

阶段 2 当前推荐的正式模型路线是：

- 轻量 `YOLO` 检测模型
- 导出为 `ONNX`
- 放置到项目的 `models/` 目录

当前工程默认约定的模型文件路径为：

- `./models/yolo11n.onnx`

说明：

- 这里的文件名是当前工程约定，不代表必须只能使用这一种模型
- 只要后续选择的模型输出格式与当前解析逻辑兼容，就可以继续使用
- 若后续改用其他轻量 ONNX 检测模型，可同步调整配置文件

当前关于模型的具体处理原则是：

- 先把检测模块和配置接入工程
- 模型文件后补
- 如果模型暂时还没有，工程仍应可正常编译运行，并明确提示当前处于无模型或 fallback 状态

当前阶段 2 暂定候选配置为：
- `frame_width/frame_height = 640x480`
- `detector_model_path = ./models/yolo11n_320.onnx`
- `detector_input = 320x320`
- `detector_interval = 2`

说明：
- 这不是最终定稿配置，而是当前阶段 2 的优先候选配置
- 之所以先定为这一组，是因为它在当前测试里兼顾了检测耗时、实际帧率和主观跟手感
- 若后续在跟踪层或摔倒规则层发现稳定性不足，再继续回看 `416` 版本或调整阈值策略

下一步性能与稳定性测试优先考虑：
- 保持上述候选配置不变
- 只调整 `detector_confidence_threshold`
- 重点观察躺地、蜷缩、局部人体等疑难姿势下的检测情况

## 6.1 Pose-based v2 规划

- `Pose-based v2` 不推倒当前工程，继续复用 `camera`、主循环和可视化框架
- 工程不推倒重来，采用 `pipeline_mode` 做配置切换
- `pipeline_mode = bbox` 时沿用当前基于框的 baseline 链路
- `pipeline_mode = pose` 时启用关键点链路
- 当前阶段仍优先使用摄像头输入，不强制先做视频离线测试
- 若 `pose` 模型尚未导入，程序应明确提示模型缺失或后端待接入，同时仍可显示摄像头原始画面
- `HOG` 继续视为 `bbox` 管线下的一个后端备选，不单独作为新管线

#### Pose 方案候选

- 方案 P1：`MoveNet Lightning`
  优点：官方资料清晰，单人 17 关键点，边缘端思路明确。
  风险：当前已验证无论官方 `TFLite` 还是第三方 `ONNX`，在树莓派现有 `OpenCV DNN` 环境下均遇到模型加载兼容性问题。
- 方案 P2：`MP-PersonDet -> MP-Pose`
  优点：来自 `OpenCV Zoo / MediaPipe Pose` 路线，更贴近当前 `OpenCV DNN` 工程，人体检测和姿态估计配套关系更清晰。
  风险：不是单模型方案，需要同时接入人体检测和姿态估计两个模型。
- 方案 P3：`YOLO Pose`
  优点：与当前 YOLO 检测工程思路接近，工具链心理成本低。
  风险：树莓派 4B 上实时性能压力更大，当前不作为第一优先级。

#### 当前 Pose 方案判断

- `MoveNet Lightning` 仍保留为已探索但暂时搁置的候选路线，不立即删除现有骨架
- 当前更值得优先推进的路线调整为：`OpenCV Zoo MP-PersonDet + MP-Pose`
- 原因是：它更符合当前 `OpenCV DNN` 工程基础，比继续深挖 `MoveNet + 当前 OpenCV 兼容性` 更稳

当前 `Pose-based v2` 的推进顺序：
- 保留当前 `MoveNet` 骨架，作为参考和备选，不删代码
- 下载并接入 `MP-PersonDet` 与 `MP-Pose` 两个 ONNX 模型
- 先完成人体关键点可视化
- 再实现基于关键点的摔倒规则判断

#### 当前 Pose v2 链路与 bbox 基线对比

- 当前 `bbox` 基线链路：
  `camera -> PersonDetector(YOLO11 ONNX) -> SinglePersonTracker -> FallDetector(box rules) -> renderer`
- 当前 `pose` 试验链路：
  `camera -> MPPersonDet -> MPPose -> renderer`

两条路线的差别：

- `bbox` 路线更轻，已经具备第一版摔倒状态机，但对躺地和怪姿势鲁棒性有限
- `pose` 路线信息量更大，理论上更适合摔倒检测，但当前性能压力明显更高，且需要更复杂的前后处理

#### 绿色人体框的当前定位

- 当前绿色人体框不是模型原生直接输出的框，而是根据 33 个关键点位置动态计算出的可视化边界框
- 现阶段它主要服务于调试和后续展示设计，例如为后续放置置信度、状态标签和摔倒结果预留视觉锚点
- 目前其实现先放在 `App` 的可视化辅助逻辑中，便于快速验证
- 若后续长期保留，建议再下沉到 `pose` 可视化工具或 `PoseFallDetector` 相关模块中，避免 `App` 继续变重

- 当前 pose 路线已经能够做到“检测姿态并输出是否疑似摔倒”，但精度、稳定性和实时性仍需继续验证

#### PoseFallDetector v1 方案

- 第一版 `PoseFallDetector` 不使用绿色人体框作为核心判断输入
- 核心输入直接来自 `MP-Pose` 的 33 个关键点与时间戳
- 第一版优先使用：
  - 躯干主轴与竖直方向的夹角
  - 髋部中心的向下运动速度
  - 基于关键点计算的身体竖向跨度相对基线变化
  - 左右膝角度，用于抑制明显下蹲/蜷缩造成的误报
  - 时间连续性状态机
- 当前状态机保持与 `bbox` 路线一致：
  - `no_target`
  - `normal`
  - `suspected_fall`
  - `fall_detected`
- 第一版目标是先验证：
  - 站立与明显倒地动作能否分开
  - 缓慢下蹲或弯腰是否能减少误触发
  - 后续再决定是否要加入更复杂的姿态时序模型

#### Pose 路线后续提速思路

- 方案 S1：继续在树莓派本地优化
  - 优点：系统结构最简单，部署最直接
  - 局限：当前 `MP-PersonDet + MP-Pose` 已经显示出明显性能瓶颈，继续单靠小调参的收益可能有限
- 方案 S2：树莓派负责采集与显示，电脑负责推理
  - 优点：最符合当前项目现状，也和队友已经验证过的思路一致
  - 适合用途：先把 pose 摔倒检测算法做稳，再考虑是否回迁到树莓派本地
  - 代价：需要增加图像传输、结果回传和两端同步逻辑
- 方案 S3：继续寻找更轻的姿态模型或更高效后端
  - 优点：若成功，可保留边缘端一体化部署
  - 风险：重新换模型或换后端会带来新的兼容性和调试成本

当前建议：
- 短期内保留树莓派本地版本作为可运行原型
- 中期优先评估“树莓派采集 + 电脑推理”的协同方案
- 等队友相关代码同步后，再决定是否复用其通信结构或目录组织方式

#### Pose 路线后续推进判断

- 当前 pose 路线的主要瓶颈已不再是“模型是否能加载”，而是：
  - 树莓派本地推理的实时性偏低
  - 关键点在复杂角度、局部遮挡、脸部细节上的稳定性仍有限
  - `PoseFallDetector v1` 仍需要继续结合真实摔倒素材调参
- 因此后续推进建议分为两层：
  - 先继续把当前本地 pose 路线打磨到“可稳定演示”的程度
  - 再评估是否仿照队友方案，引入“树莓派采集 + 电脑端推理”的提速路线
- 如果后续引入电脑侧算力，当前最自然的思路是：
  - 树莓派继续负责摄像头采集与轻量显示
  - 电脑端负责较重的人体检测 / 姿态估计 / 摔倒判定
  - 两端之间通过网络传输视频帧或中间结果
- 该路线的优点是：
  - 能显著缓解树莓派本地推理压力
  - 更容易尝试更重但更稳的 pose 模型
  - 便于后续把 bbox / pose / 其他模型方案统一到电脑端做对比
- 该路线的风险是：
  - 系统复杂度上升
  - 需要处理网络传输延迟、同步、协议与容错
  - 需要先参考并理解队友现有的电脑协同推理实现

## 7. 当前结论

当前已经确认的方向如下：

- 使用树莓派 4B 和 USB 摄像头
- 开发语言以 C++ 为主
- 第一版限定为固定机位、室内场景、单人检测
- 第一版输出以屏幕可视化为主
- 技术路线优先选择“现成轻量模型 + 规则判定”
- 开发顺序采用“摄像头 -> 人体检测 -> 姿态/关键点 -> 摔倒判断”
- 实现策略采用“我们自己写，但主动对齐队友工程风格和接口习惯”
- 阶段 2 正式路线优先采用 `OpenCV DNN + ONNX`
- 当前运行入口采用“主菜单选择高层模式 + `app.yaml` 保存细参数”的方式：
  - `HOG + Box`
  - `YOLO11 + Box`
  - `MediaPipe + Pose`

## 8. 主要风险

### 风险 1：树莓派性能有限

人体检测和姿态估计都可能带来明显卡顿，因此需要控制分辨率、帧率、检测间隔和模型大小。

### 风险 2：室内场景差异较大

宿舍、实验室、教室的背景、光照和摆放方式不同，规则判断可能需要调参。

### 风险 3：非摔倒动作的误报

蹲下、坐地、弯腰、捡东西都有可能被误判为摔倒，因此不能只用单帧姿态下结论。

### 风险 4：关键点稳定性不足

如果人体被遮挡、距离过远、画面偏暗，关键点检测效果可能下降。

## 9. 下一步建议

当前建议优先推进以下事项：

1. 继续保留当前 `bbox baseline` 与 `MediaPipe + Pose` 本地版本，作为可运行基线和后续对照
2. 在当前阶段，将“树莓派采集 + 电脑侧推理”确定为下一条重点探索路线
3. 第一优先采用 `PC Pose 方案 A`：
   - 树莓派侧负责摄像头采集、画面显示、异步上传当前整帧图像
   - 电脑侧负责姿态模型推理，返回 33 个关键点结果
   - 树莓派侧继续复用现有关键点绘制逻辑与 `PoseFallDetector`
4. `PC Pose 方案 A` 的第一阶段先不接真实模型，先做“假服务 + 联通测试”：
   - 电脑侧先提供 `/health` 和 `/analyze_pose` 两个最小接口
   - `/analyze_pose` 先返回固定 JSON，验证 Pi -> PC -> Pi 的请求与回传链路
   - 当前运行入口对应主菜单 `4) MediaPipe (PC) + Pose (Pi)`
   - 第一阶段默认只在内存中接收图像并立即返回结果，不默认落盘保存请求图片
   - 联通确认后，再把电脑侧替换成真实 `MediaPipe Pose` 推理
5. `PC Pose 方案 A` 的第二阶段采用官方 `MediaPipe Pose Landmarker`：
   - 电脑侧使用 Python `Pose Landmarker` 服务，并优先选用 `Lite` 模型
   - 电脑侧采用 `VIDEO` 运行模式，输入为树莓派上传的整帧 JPEG
   - 接口仍保持 `/health` 与 `/analyze_pose`，尽量不改动树莓派端通信协议
   - 电脑侧返回 `33` 个关键点与基础分数信息，树莓派侧继续复用现有关键点绘制和 `PoseFallDetector`
6. 在通信链路稳定前，不建议继续大幅增加树莓派本地 pose 管线复杂度，避免继续恶化实时性
