# 摔倒检测项目方案草案 v1

## 1. 项目目标

本项目拟使用树莓派 4B 和 USB 摄像头，完成一个面向学校室内环境的单人摔倒检测原型系统。

当前第一版目标如下：

- 场景限定在室内，如宿舍、实验室、教室
- 摄像头采用固定机位
- 只考虑单人场景
- 第一版输出以屏幕可视化为主，不先做报警联动
- 开发语言以 C++ 为主

这个版本的重点是先把完整链路跑通，而不是一开始追求复杂场景下的高精度。

## 2. 第一版边界

第一版计划做的内容：

- 采集 USB 摄像头视频流
- 在树莓派上实时显示画面
- 完成人体检测与框选
- 在条件允许时增加人体关键点或姿态可视化
- 基于规则做摔倒判断
- 在屏幕上显示当前状态，如 `normal`、`suspected fall`、`fall detected`

第一版暂不优先做的内容：

- 多人同时检测
- 复杂遮挡场景优化
- 自训练摔倒分类模型
- 语音报警、短信推送、联网通知
- 跨场景高泛化能力

## 3. 总体技术路线

当前更适合的路线不是直接训练一个完整的摔倒识别模型，而是采用“现成轻量模型 + 规则判定”的方案。

推荐拆分如下：

1. 视频采集与显示：使用 OpenCV 完成摄像头读取、画面显示和截图调试
2. 人体检测：使用轻量人体检测模型，先确认树莓派 4B 上的运行速度是否可接受
3. 姿态估计：在人体检测稳定后，再评估是否加入关键点检测
4. 摔倒判定：结合人体框比例、身体朝向、关键点高度变化、持续时间等规则进行判断

这样做的原因：

- 比自己训练模型更容易落地
- 更适合树莓派 4B 的算力条件
- 能够分阶段验证，不容易一开始就陷入模型训练和部署细节

## 4. 对齐队友工程的原则

为了便于后续和队友的人脸识别项目保持风格一致，并为未来合并预留空间，当前采用以下策略：

- 保持目录层次和模块划分风格尽量一致
- 参考对方的入口组织方式和类命名风格
- 优先对齐通用模块接口，如相机、渲染、配置读取方式
- 不直接复制与人脸识别强绑定的业务逻辑
- 在保证功能清晰的前提下，为后续合并预留兼容空间

实现策略明确为：

- 我们自己写
- 主动对齐队友工程风格和接口习惯
- 不直接整套移植参考工程

## 5. 分阶段开发计划

验收标准：

- 程序启动后可持续显示摄像头画面
- 能正常退出，不出现频繁卡死

当前阶段 1 已完成，已实现内容如下：

- 已搭建最小 C++ + CMake 工程骨架
- 已按约定保持与队友工程相近的入口和模块风格
- 已完成 USB 摄像头采集与实时显示
- 已完成树莓派端依赖检查和 CMake 配置验证
- 已确认树莓派能够识别当前 USB 摄像头设备

当前阶段 1 的实际结论：

- 摄像头链路已经打通
- 当前工程可以在树莓派上正常编译和运行
- 后续可以进入阶段 2，即人体检测与框选

### 阶段 2：人体检测与框选

目标：

- 在视频中识别人
- 用矩形框标记人体区域
- 在画面上显示基础检测信息
- 第一版只保留 `person` 类别
- 第一版在单人模式下只保留一个主框

验收标准：

- 单人站立、走动、弯腰时能较稳定框住人体
- 树莓派端运行速度基本可接受
- 缺少模型文件时，工程仍可编译运行并给出明确状态提示

#### 阶段 2 当前选定方案

当前阶段 2 的正式路线为：

- 推理框架：`OpenCV DNN`
- 模型格式：`ONNX`
- 第一目标类别：`person`
- 运行模式：单人场景下保留一个主框

选择这条路线的原因：

- 已有工程已使用 `OpenCV + CMake + C++`
- 接入成本低于直接引入 `ncnn`
- 便于先把人体检测功能跑通
- 后续若树莓派性能不足，仍可把检测后端替换为 `ncnn`

#### 阶段 2 详细推进步骤

1. 先增加独立的 `PersonDetector` 模块，并接入工程
2. 在配置文件中加入模型路径、输入尺寸、阈值、检测间隔等配置项
3. 在主循环中接入人体检测，并在画面上绘制人体框和置信度
4. 在单人模式下，只保留一个主目标，减少后续逻辑复杂度
5. 增加基础性能控制手段，如检测间隔和结果复用
6. 在树莓派上测试检测稳定性、延迟和分辨率影响
7. 根据实际效果决定是否调低输入分辨率或降低检测频率

#### 阶段 2 当前工程策略

为了便于阶段 2 尽快落地，当前工程层面采用以下策略：

- 正式路线使用 `YOLO ONNX + OpenCV DNN`
- 在模型文件尚未准备好时，保留一个轻量 fallback，避免工程完全不可运行
- 当前阶段先做检测和画框，不做跟踪、不做关键点、不做摔倒判定

- 当前已经进入正式的 `ONNX + OpenCV DNN` 路线
- 接下来阶段 2 的重点从“能否检测”转为“性能调优与参数平衡”

#### 阶段 2 当前性能调优方向

当前关于帧率问题的结论如下：

- `frame_fps` 表示向摄像头申请的目标帧率，不代表最终程序实际显示帧率
- 实际 FPS 会受到摄像头取流、模型推理耗时、检测间隔和分辨率等多个因素影响
- 当前优先采用小步调优，而不先引入云端协同等复杂方案

当前阶段 2 第一轮性能优化的策略：

- 在界面中显示实际 FPS 和单次检测耗时
- 优先调低检测输入尺寸，而不是立刻降低显示分辨率
- 适当增加检测间隔，降低每秒推理次数
- 在保证单人框选效果基本可用的前提下，逐步平衡准确率和流畅度

### 阶段 3：姿态或关键点可视化

目标：

- 在画面中绘制人体关键点
- 用不同颜色线条连接关节点
- 观察不同姿态下的稳定性

验收标准：

- 站立、弯腰、坐下、倒地时关键点有基本可视化效果

风险提示：

- 这一阶段比人体检测更吃性能
- 若树莓派运行过慢，可将其降级为可选模块，或延后实现

### 阶段 4：摔倒判定

目标：

- 输出正常、疑似摔倒、确认摔倒三种状态
- 不只依赖单帧结果，而是结合短时间窗口进行判断

建议规则来源：

- 人体框长宽比变化
- 身体主方向接近水平
- 头部、肩部、髋部高度变化明显
- 姿态变化在短时间内发生
- 倒地状态持续一定时间

验收标准：

- 对明显摔倒动作能给出可视化提示
- 对普通站立和缓慢行走不频繁误报

#### 摔倒检测方案对比

- 方案 A：基于框几何特征 + 时间规则
  当前主推方案。直接复用现有人体检测结果，工程量最小，最适合先做出可运行雏形。
- 方案 B：单人跟踪 + 几何规则 + 丢框保持
  当前推荐落地方案。相比方案 A，更能应对短时掉框、低置信度和姿势突变，适合作为阶段 4 的直接前置能力。
- 方案 C：姿态关键点方案
  可作为后续增强方向。理论上更适合分析摔倒姿态，但对树莓派 4B 的性能压力更大，暂不作为第一优先级。
- 方案 D：直接 fall / stand 分类模型
  暂不优先。数据集、训练与部署复杂度更高，容易把项目重心过早拉向模型调优。

#### 当前推进顺序

- 先做单人框跟踪、低置信度延续和丢框保持
- 再做基于框的摔倒规则判断
- 姿态关键点和直接分类模型作为后续增强路线，暂不先上

后续实现备注：
- 若后期需要并行保留多种检测方法，可优先采用“配置切换后端/模型路径”的轻量方式，而不是在运行主链路里加入复杂交互
- 当前阶段先保证单一路线可稳定运行，避免为了切换功能影响实时效果

状态机说明：
- 队友项目中的 `core/StateMachine` 更偏向应用流程状态切换，不直接复用到本项目的摔倒判定逻辑
- 当前项目在 `FallDetector` 内部维护 `normal / suspected_fall / fall_detected` 这一类领域状态，更贴合当前单人摔倒检测场景

## 6. 模型准备与下载说明

阶段 2 当前推荐的正式模型路线是：

- 轻量 `YOLO` 检测模型
- 导出为 `ONNX`
- 放置到项目的 `models/` 目录

当前工程默认约定的模型文件路径为：

- `./models/yolo11n.onnx`

说明：

- 这里的文件名是当前工程约定，不代表必须只能使用这一种模型
- 只要后续选择的模型输出格式与当前解析逻辑兼容，就可以继续使用
- 若后续改用其他轻量 ONNX 检测模型，可同步调整配置文件

当前关于模型的具体处理原则是：

- 先把检测模块和配置接入工程
- 模型文件后补
- 如果模型暂时还没有，工程仍应可正常编译运行，并明确提示当前处于无模型或 fallback 状态

当前阶段 2 暂定候选配置为：
- `frame_width/frame_height = 640x480`
- `detector_model_path = ./models/yolo11n_320.onnx`
- `detector_input = 320x320`
- `detector_interval = 2`

说明：
- 这不是最终定稿配置，而是当前阶段 2 的优先候选配置
- 之所以先定为这一组，是因为它在当前测试里兼顾了检测耗时、实际帧率和主观跟手感
- 若后续在跟踪层或摔倒规则层发现稳定性不足，再继续回看 `416` 版本或调整阈值策略

下一步性能与稳定性测试优先考虑：
- 保持上述候选配置不变
- 只调整 `detector_confidence_threshold`
- 重点观察躺地、蜷缩、局部人体等疑难姿势下的检测情况

## 7. 当前结论

当前已经确认的方向如下：

- 使用树莓派 4B 和 USB 摄像头
- 开发语言以 C++ 为主
- 第一版限定为固定机位、室内场景、单人检测
- 第一版输出以屏幕可视化为主
- 技术路线优先选择“现成轻量模型 + 规则判定”
- 开发顺序采用“摄像头 -> 人体检测 -> 姿态/关键点 -> 摔倒判断”

- 阶段 2 正式路线优先采用 `OpenCV DNN + ONNX`

## 8. 主要风险

### 风险 1：树莓派性能有限

人体检测和姿态估计都可能带来明显卡顿，因此需要控制分辨率、帧率、检测间隔和模型大小。

### 风险 2：室内场景差异较大

宿舍、实验室、教室的背景、光照和摆放方式不同，规则判断可能需要调参。

### 风险 3：非摔倒动作的误报

蹲下、坐地、弯腰、捡东西都有可能被误判为摔倒，因此不能只用单帧姿态下结论。

### 风险 4：关键点稳定性不足

如果人体被遮挡、距离过远、画面偏暗，关键点检测效果可能下降。
